Event System:
The event system was created to slowly build up tension with dynamically spawning scares. An example of a low level event is the log, which spawns a floating log with a pointing up branch that could be mistaken as Nessie.

A challenge with the event system was what Unreal class was perfect for the job, considering it had to replicate the events to all players in this co-op game. After some trial and error I came to the conclusion a component on the boat fit the bill, as it was a pawn controlled by the player it allowed server RPCs from the client. We only wanted the events to fire whilst the player was on the water so the boat being the center of event spawning, not only made sure it replicated but made sense from a design perspective.

It was fun getting creative. A personal favourite event is the peaking nessie event where the nessie would stick her head out of the water by the player's boat, stare at them for a duration and if spotted would submerge back into the water playing an eerie sound.

Through developing the event system I learned the difficulties of utilizing randomness in game development and the importance of the hierachy the Unreal Engine classes (especially when it comes to multiplayer).

Replication:
Any replication in the project, or any bugs (which you can imagine was many) was handled predominately by me. Anytime our designer made a cool new mechanic, say fishing, it was up to me to refactor the code to not only fix any bugs but also for the code to work in multiplayer. This was one of the biggest challenges of Loch Ness, the designer would work soley with singleplayer in mind and I would have to step in to convert it to multiplayer.

To name some of the biggest challenges with converting singleplayer gameplay into multiplayer gameplay was our pong minigame, our dead by daylight inspired minigames, that if failed could attract nessie, the boat movement on water and the map showing multiple players orientation to name a few.

This experience not only taught me how to refactor singleplayer code and convert it to multiplayer. But, how to communicate potential issues with design to my team.

Pong:
Pong seems like such a simple game, but to replicate this over the network was a nightmare. Widgets in Unreal are client only, this combined with the fact the designer created it with only singleplayer in mind made it a very hard problem to solve. It required a major refactor, and to get it to feel exactly the same as the designer intended was difficult to say the least. Luckily there are ways to replicate data for all connected players in widgets, by each client agreeing on the state of the game through centering the data retrieved on the game through the GameState class in Unreal.

In retrospect working on Pong in Loch Ness taught me not to be afraid trying new things and research into the software tools you're using goes a long way.

AI:
At the start of each game Nessie spawned randomly at one of the sighting locations which was also random.

The Nessie would wander the sighting location using steering behaviours, staying within the proximity of the sighting location.

Nessie would only emerge chasing the player if they triggered her, either from certain events or through the various minigames whilst collecting evidence.

So to conclude the AI, it used the state pattern to have control over current behaviour and steering behaviour to have natural looking movement.

Using the state pattern taught me that having too many exit conditions from a state made it incredibly hard to debug and control.

Conclusion:
Loch Ness was our Tranzfuser project, it was our first major project as a team. For our first game it did well at over 3000 units sold and over 5000 wishlists. It fell short because of some issues with bugs and performance over the course of the game. Ultimately, we ran out of funding for the game and had to move on because of of another opportunity (Dun Dev).

It taught us how to work better as a team, as well as, what to avoid in development like excessive use of physics and randomness in multiplayer.

Personally I learned a bunch about Unreal, structuring code, multiplayer, optimization, AI, testing and game development as a whole, whilst working in a small team.
